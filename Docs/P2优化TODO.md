# P2 优化 TODO（内存与性能）

本文档聚焦 P2 阶段优化，目标是在已完成 P0/P1 的基础上，进一步降低长会话场景的内存占用并提升滚动与交互流畅度。

## 目标与范围

- 目标 1：长会话详情页在前端保持稳定的内存占用。
- 目标 2：减少后端列表接口的 N+1 查询开销，降低 CPU 和临时对象创建。
- 目标 3：优化大文本传输与渲染策略，减少峰值内存和主线程阻塞。

## P2-1: 详情消息虚拟列表（前端）

### 背景

当前详情页会直接渲染已加载的全部消息，随着分页累计，DOM 节点和 React Fiber 数量持续增长，导致内存与重排成本上升。

### TODO

- [ ] 选型并接入虚拟列表库（建议 `react-window`）。
- [ ] 将 `detail.messages.map(...)` 改为虚拟渲染，仅保留可视区附近节点。
- [ ] 为消息项提供稳定高度策略：
  - [ ] 首版采用估算高度 + overscan。
  - [ ] 对代码块/折叠块变化触发局部重测。
- [ ] 保持现有功能不回退：
  - [ ] 悬浮快捷导航（回顶、上/下一个 prompt）。
  - [ ] part 的按需加载与展开。
- [ ] 加入空态/加载态兼容，避免虚拟列表容器高度抖动。

### 验收标准

- [ ] 同一会话加载 1000+ 条消息后，前端内存不再随消息数量线性增长。
- [ ] 滚动 FPS 在常见机器上稳定，无明显掉帧。

## P2-2: 列表预览去 N+1 查询（后端）

### 背景

`getSessions` 目前为每个 session 分别查询预览 part，存在 N+1 问题，在列表页请求时会放大 SQL 扫描与对象创建。

### TODO

- [ ] 将当前页 session id 收集为批量集合。
- [ ] 使用单次批量 SQL 拉取预览所需 part（按 session 分组后截断）。
- [ ] 在内存中按 session 聚合并生成 preview，替换逐条查询。
- [ ] 保留现有排序与分页语义，不改变接口返回结构。

### 验收标准

- [ ] `getSessions` 的 SQL 调用次数从 O(n) 降为 O(1) 级别（每页固定少量查询）。
- [ ] 列表接口平均响应时间与 CPU 占用显著下降。

## P2-3: 大文本截断与延迟展开（后端+前端）

### 背景

部分 part（tool output/patch/file）内容很大，即使按需拉取，也可能在单次展开时造成内存峰值和渲染卡顿。

### TODO

- [ ] 在 part 详情接口增加统一截断策略：
  - [ ] 文本长度上限（例如 100KB，可配置）。
  - [ ] 标记字段 `truncated: true` 与 `originalSize`。
- [ ] 前端在 `CodeBlock` 展示“已截断”提示与二次加载入口。
- [ ] 对超大内容默认按分段加载（可选 cursor/chunk 方案）。

### 验收标准

- [ ] 展开超大 patch/file 时，前端无明显卡死。
- [ ] 峰值内存较未截断方案明显下降。

## 监控与验证

- [ ] 浏览器侧：使用 Performance + Memory 记录 10 分钟连续滚动与展开操作。
- [ ] Node 侧：记录 RSS/heapUsed 曲线，确认进入平台期。
- [ ] 回归测试：
  - [ ] 搜索/列表分页正常。
  - [ ] 详情分页与 part 懒加载正常。
  - [ ] 中英文与主题切换不受影响。

## 实施顺序（建议）

1. 先完成 P2-2（后端去 N+1，风险低、收益稳定）。
2. 再做 P2-1（虚拟列表，收益最大）。
3. 最后落地 P2-3（超大内容策略，完善极端场景）。
